為什麼要寫技術文章？

取之於社會，用之於社會。

年幼時曾經營過 Blog，不過重點著重在軟體分享，除了需要龐大的營運成本（寫文、提供空間、更新版本軟體）。另外分享軟體好像也就如此，而市面上分享軟體太多了，不如就交給更專業的吧。

但日常生活中，重複講解的技術太多，能抱怨的事情太多，能分享的事情太多。

不如就把它紀錄下來，試著與大家分享心得與觀點。

為什麼選用 Hugo 當作框架？

在開始撰寫之前，以下是我個人考慮的因素

  1. 無伺服器 (Serverless)

    建置傳統的 WordPress 走的是 PHP 框架，需要伺服器進行後端運行。相較於靜態網頁，可能找個 GitHub Pages, Cloudflare Pages, Firebase Hosting 之類的丟上去就完事了。

  2. 速度 (Speed)

    網頁預覽速度要快，沒有太多的外掛、廣告。

  3. 易用性 (Usability)

    產生靜態網頁要簡單、避免自己在幾個月後忘記還要翻 API 研究如何產出靜態目錄、更新套件等一系列操作。

  4. 人氣 (Popularity)

    一個框架如果人氣不夠，未來的可擴展性可能就被侷限住了。

市面上常見的網頁框架主要包括 Jekyll、Hexo 和 Hugo。不過考量各種因素後，最終選擇了 Hugo。

最初的選擇實際上是 Hexo，這是一個建立在 Node.js 上的框架。安裝完 Hexo 後就能夠開始撰寫文章並進行編譯。

然而，在2023年9月寫下第一篇文章後，直到2023年11月才真正開始撰寫。由於對語法不夠熟悉，花了3分鐘回憶。

此外，Node.js 與 Hexo 各自有特定的版本需求，使用 npm outdated 和 npm update 可以對 Hexo 的檢查版本與更新，但也間接地形成軟體維護更新的成本。 （Node.js (20.9.0 LTS)、Hexo (7.0.0)）

相比之下，Hugo 是透過 Go 開發的，是一個執行檔，安裝後一切就緒，進入到部落格的資料夾後下指令就可以建立 Server 或是編譯，使用上真的是快。

對於其他平台像是：

  1. Jekyll 則被聽說多文章編譯會較慢，而且安裝還需要下載 Ruby，類似 Node.js 的操作行為我也就沒有繼續深入探討
  2. VuePress 相對小眾，從推廣的程度上來看似不及 Hexo 和 Hugo。

不過無論如何，文章格式都是 Markdown，未來如果需要在不同平台轉換，應該是無感知的。在現階段要開始撰寫技術文章，選擇一個使用者眾多、具有成長性的框架就足夠了。

結論

本部落格為撰寫技術文章、分享心得，如果看到文章覺得有幫助也歡迎轉發網址給更多人看，若內容有誤也請不吝嗇回覆給我，非常感謝。
